# huck 0.3.0

The largest release yet — roughly 70 numbered iterations (v212 → v284) since
0.2.0. The headline is a ground-up rebuild of the language front-end into a
clean lexer/parser split, but the release also brings a POSIX-mode foundation,
a wave of builtin and bash-compatibility work, a consolidated public crate API,
and a marketing/documentation website.

As of this release the workspace crates (`huck-cli`, `huck-engine`,
`huck-syntax`) are version-aligned with the `huck` binary — all four are `0.3.0`.

## Headline: the parser-driven front-end re-architecture (v238 → v268)

huck's original front-end had a "fat lexer" that forward-scanned for delimiters
— matching `$( )`, `${ }`, `` ` ` ``, `(( ))`, heredocs, and array literals
inside the tokenizer itself, then handed a mostly-assembled token stream to the
parser. That design made every quoting/nesting edge case a lexer problem and
left two front-ends (a legacy "oracle" and the new path) to keep in sync.

Over ~30 iterations that architecture was replaced wholesale. The end-state,
now live and the sole front-end:

- **The lexer emits small atoms and never forward-scans for a delimiter.**
- **The parser owns all delimiter-matching, recursion, and word/structure
  assembly**, driving the lexer's tokenization mode through a mode stack.

Milestones in the arc: incremental pull-based lexing (v238) and parser-pulls-live
(v239); a lexer mode stack with mark/rewind (v240); parser-driven parameter
expansion, command lists, and compound commands (v241–v243); command
substitution, backtick, and arithmetic sub-modes (v244–v246); command atoms,
function definitions, here-strings, heredocs, process substitution, and array
literals (v247–v252); `[[ ]]`, `=~` regex, `(( ))` and arith-`for` (v253–v256);
`coproc` and legacy `$[ ]` arithmetic (v257–v258); carry-forward reconciliation
and heredoc/quote sub-mode fixes (v259–v263); flipping the atom path live
(v264), **deleting the legacy oracle entirely** (v265–v266), bounding lexer
history lookahead (v267), and severing the last lexer→parser cycle (v268).
v279 removed the final forward-scanner (`${…}` bad-substitution is now assembled
by the parser from a zero-width marker), leaving the lexer forward-scanner-free.

The payoff is a front-end that is smaller, easier to reason about, and more
bash-faithful at the quoting/nesting edges — verified by a parser coverage sweep
(`tools/parse_sweep.sh`) against both a 4600-script local corpus and the bash
5.2.21 test suite.

## POSIX mode foundation (v213 → v237)

A `set -o posix` foundation and the behaviors that hang off it:

- **`set -o posix` core** and mode plumbing.
- **POSIX exit-on-error** semantics and `set -e` refinements.
- **`command`-position alias handling**, `source`/alias fixes, and
  command-not-found word-order correctness.
- **`getopts` error-reporting** aligned to bash's prologue.
- **`umask` / `ulimit` / `enable`** builtin coverage.
- **IO-error text no longer leaks** into captured output.
- **`$"…"` / `$'…'` extended-quote** parsing hardened across quote contexts,
  including indirect `${!x}` interaction.

## Builtins and bash compatibility

- **`read` cluster** (v276) — `-n` / `-N` / `-t` (char count, exact count,
  timeout).
- **`history` builtin** (v284) — `history N` plus `-d` / `-w` / `-r` / `-a`
  (delete by offset/negative/range, write, read, append-new) on top of the
  existing `-c`. Closes #6 and #7.
- **`export` of arrays** (v282) — `export a=(...)` assigns and exports via the
  normal path; exported arrays no longer leak `[0]` into the child environment.
  Closes #82 and #28.
- **`set -e` / ERR on non-final `&&`/`||`** (v275) — a real behavioral-bug fix.
- **Script parse errors report the correct line** at a compound-unit boundary
  and no longer drop earlier commands (v277). Closes #86.
- **Comment-only / empty `$( )` and `<( )` bodies at EOF** now parse correctly
  from stdin instead of erroring (v281). Closes #109.

## Public crate API

- **Tier 1 surface cleanup** (#100) — hid engine-internal and CLI-only modules,
  renamed `Engine::exec` → `prepare`, and surfaced `Lexer` / `parse_sequence` /
  `parse` from `huck-syntax`.
- **Tier 2 consistency polish** (#103) — `#[non_exhaustive]` on engine return
  types, self-contained `huck-syntax` root re-exports, `Redirect` → `RedirectSlot`,
  and `restricted()` / `version()` knobs on the builder.
- **Unified error emitter** (v269) — all ~374 error sites route through a single
  `sh_error!` / `emit_error` prologue (thread-local plus writer-based variants),
  fixing error-text leakage under `$(cmd 2>&1)` capture.
- **`Lexer::new`** constructor collapse (#95) and a **macOS build/test fix**
  (#96/#98).

## Website and distribution

- A **marketing + blog website** (Next.js on Vercel) with a **Library page**
  documenting the `huck-syntax` / `huck-engine` crate APIs (#88, #92).
- Distribution remains self-hosted off GitHub Releases: the `.deb` package, the
  Homebrew tap `jdstanhope/homebrew-huck`, and the `curl | sh` installer.

## Test and CI hardening

- The **`tests/scripts/*_diff_check.sh` sweep now runs in CI** via a committed
  `run_diff_checks.sh`, each harness on its own default binary (v280).
- Inline `#[cfg(test)] mod` blocks were **extracted into 57 per-module sibling
  test files** (v278), a pure file-decomposition refactor.
- The bash-vs-huck diff harness count and the parser-coverage sweeps continued to
  grow; the front-end rework is guarded end-to-end by byte-identical harnesses.

## Compatibility notes

- **Workspace crate versions are now aligned** with the release version. The
  crates are still path dependencies and are not published to crates.io.
- **Public API changes** in `huck-engine` (`exec` → `prepare`, `#[non_exhaustive]`
  return types) and `huck-syntax` (`Redirect` → `RedirectSlot`, curated
  re-exports) are breaking for embedders pinned to 0.2.0 — hence the minor bump.
- The front-end rework is behavior-preserving toward bash by design; a small set
  of exotic, kept-by-design divergences is tracked in `docs/bash-divergences.md`.
